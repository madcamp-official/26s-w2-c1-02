# STT 클라이언트 미니 워크플로우 — 청크 분할 + 타임스탬프 병합

> [workflow.md](../workflow.md) Step 2 팀원3 항목 "STT 클라이언트(최난도)"의 세부 계획.
> 작성일: 2026-07-11 · 담당: 팀원3

## 계약 (입출력)

- **입력**: 발표 녹음 파일 경로 (mp3/wav/m4a, ≤60분·200MB — spec §1.3)
- **출력**: `[{"start": 12.0, "end": 15.2, "text": "..."}]` — 초 단위 float, `transcripts.segments` JSONB 그대로 (db-schema §6.2). API 응답의 `ts:"MM:SS"` 문자열 변환은 팀원2 라우터 몫
- **호출 형태**: `services/stt.py`의 동기 함수 1개 — 팀원2의 STT 잡(큐)에서 호출

```python
def transcribe_recording(audio_path: str | Path, *, language: str = "ko") -> list[dict]:
    """긴 녹음 → 청크 분할 → STT 직렬 호출 → 오프셋 합산 병합. 실패 시 SttError."""
```

- **서버 계약** (stt_server.py, :8200): `POST /transcribe` multipart(file, language, timestamps) → `{"text", "segments": [{"text","start","end"}]}`. 제약: **1건당 ≤5분(ForcedAligner)** · **직렬 처리**(동시 요청 금지)

## 청크 파라미터 — Day 3 결정 재검토 (재실측 불필요)

Day 3 실측(README [STT 실측 표](../../infra/gpu-server/README.md#stt-실측--청크-크기-결정-day-3-팀원3)): RTF≈0.03~0.05, 60s→2.1s, 300s→9.7s.

재검토 사유: A8(클라이언트 로컬 녹음 → **종료 후 일괄 업로드**)이라 "청크 주기 진행률" 근거가 약해짐. 대안인 **300s 청크**(경계 12개/60분, 병합 부담 ↓)와 비교:

| | 60s+4s (유지) | 300s−α |
|---|---|---|
| 60분 총 STT | ~60×2.1s ≈ 2.1분 | ~12×9.7s ≈ 1.9분 (비슷) |
| 병합 경계 수 | 59 | 11 |
| 청크 실패 재시도 비용 | 60s어치 | 5분어치 |
| 캡 여유 | 64s ≪ 300s 안전 | 304s > 캡 → 296s로 줄여야 (캡 판정 모호) |

**결론: 60s + 4s 유지.** 총 소요 동급, 실패 격리 유리, 캡 경계 리스크 없음. 단 `CHUNK_SEC`/`OVERLAP_SEC`를 모듈 상수로 두어 한 줄 변경 가능하게 (경계 병합 품질이 나쁘면 300s−α로 전환).

## 작업 단계

- [x] **1. 오디오 정규화** — ffmpeg로 mp3/m4a/wav → 16kHz mono wav 통일. 디코드 실패 → `UNSUPPORTED_MEDIA` 매핑용 예외. ffmpeg 미설치 환경 대비 `imageio-ffmpeg`(정적 바이너리) 폴백
- [x] **2. 분할기** — 청크 i = `[i·60, i·60+64)`초 (마지막 청크는 남은 만큼). 오디오 ≤64s면 분할 없이 단일 요청
- [x] **3. 직렬 전송** — 청크 순서대로 `POST /transcribe` (서버가 직렬이므로 동시 전송 금지). 청크 단위 재시도(지수 백오프 2회), 최종 실패 → `SttError`(→ 팀원2가 `STT_FAILED`/retry)
- [x] **4. 오프셋 합산** — 각 세그먼트 `start/end += i·60` (**64가 아님** — 겹침은 stride에 안 들어감)
- [x] **5. 겹침 병합(dedup)** — 규칙: 경계 `b = (i+1)·60`에 대해 절단점 `cut = b + 2`(겹침 4s의 중앙). 앞 청크는 중점 `< cut`, 뒤 청크는 `≥ cut`인 것만 채택. **단어(형태소) 단위로 dedup 후 그룹화** (아래 발견 사항)
- [x] **6. E2E 검증** — TTS 합성 2.7분 한국어 발표(39문장, 경계 2개) 실측(2026-07-11): 전사 5.4s(RTF 0.033), 세그먼트 27개(문장급), 경계 중복/탈락 0, **CER 0.96%**(표기 정규화 제외) — Day 3 기준선(0.8%)과 일치
- [ ] **7. 팀원2 합류 검증** — JSONB 저장 왕복 + `ts` 변환 확인 (workflow Step 2 세 번째 항목과 동일 세션에서). **테스트 준비됨**: `tests/test_transcript_roundtrip.py` — `ts` 변환 7개 통과, JSONB 왕복 2개는 dev DB에서 실행만 하면 됨. 라우터는 `services/stt.py`의 `seconds_to_ts()` import해서 사용

## 구현 중 발견 사항 (설계 반영됨)

1. **ForcedAligner 스탬프는 문장이 아니라 형태소 단위** — '발표'+'를' 식으로 쪼개지고 무구두점, 무간격 정보. 그대로 저장하면 세그먼트 430개/2.7분 + 띄어쓰기 붕괴. → 응답의 punctuated `text`에 스탬프를 문자 정렬해 표기(띄어쓰기·구두점)를 물려받고, **쉼 ≥0.35s** 기준으로 문장급 그룹화(`GROUP_GAP_SEC`, 실측: 어절 내부 갭 ≤0.24s vs 문장 경계 ≥0.32s). 정렬 실패 시 공백 조인으로 강등
2. **언어 파라미터는 ISO 코드가 아니라 전체 명칭** — `ko` → 500, `Korean` 요구 (Qwen3-ASR). 클라이언트에서 `ko→Korean` 매핑
3. **ASR 표기 정규화** — '피디에프'→'PDF', '서른 배'→'30배' 등으로 출력됨(오류 아님). 리포트 지표(WPM·필러) 계산 시 참고

## 엣지케이스

| 케이스 | 처리 |
|---|---|
| 녹음 ≤64s (답변 STT 포함) | 청크 1개, 병합 스킵 — 답변 전사도 같은 함수 재사용 |
| 마지막 청크가 매우 짧음 (<4s) | 이전 청크가 이미 커버 → 잔여 <1s면 청크 생략 |
| 무음 청크 | `segments: []` 정상 — 병합 시 빈 배열 허용 |
| 겹침 구간에 걸친 긴 세그먼트 | 중점 규칙으로 한쪽에만 귀속 (자르지 않음) |
| STT 서버 다운 | 재시도 소진 → `SttError`, 부분 결과 버림 (멱등 재실행 가능해야) |

## 예상 작업량

1~4는 반나절, **5(병합)와 6(검증)이 나머지 전부** — "최난도"의 실체는 경계 dedup 품질. 6에서 경계 중복이 반복되면 겹침을 3~5s 범위에서 조정(한 줄).
